# launch 分层规则

`thesis_ws/launch/` 是 thesis 侧的正式入口层，按职责分成三层：

- `platform/`：平台接入层。只负责桥接平台现有能力，不把上游源码搬进 thesis。
- `scenarios/`：实验运行层。面向“建图展示、单点导航、多点巡检雏形”三类任务组织入口。
- `tools/`：可视化/录包层。负责 RViz、rosbag、辅助工具的 thesis 侧封装。

## 运行路径原则

- 本地 `catkin_ws` 是平级平台工作空间，不是 thesis_ws 需要嵌入进去的目录前提。
- 后续 launch 应依赖 ROS 包解析、包名和已 source 的环境，而不是依赖任何旧的 `reference` 相对路径。
- 工控机部署时，默认面向 `~/catkin_ws` 与 `~/thesis_ws` 的平级关系。

## 为什么不能直接沿用官方耦合 launch 作为 thesis_ws 正式入口

- 官方 launch 以“把功能跑起来”为主，经常把底盘、传感器、算法、RViz、固定地图路径揉在同一个文件里。
- 毕设需要的是“可展示、可复现、可记录”的实验流程，而不是单次 demo 入口。
- thesis_ws 需要自己控制 bag 命名、结果目录、地图输出位置、任务文件和 overlay 参数。
- 如果直接把官方耦合 launch 当 thesis 正式入口，后续建图、导航、巡检三条展示链会持续和平台细节缠在一起。

因此，第一轮先把 thesis 入口命名和分层定下来，即使部分场景 launch 仍是占位，也不让上游平台结构直接成为论文系统的“门面入口”。
